# ADR-0003: URL-Based Multi-Tenant Architecture with Organisation-Scoped Resources

## Context

The application is evolving from a single-tenant system into a multi-tenant Software-as-a-Service (SaaS) platform. The goal is to allow multiple independent organisations to use the same application while ensuring complete logical separation of their data.

Every business entity—including tasks, projects, files, comments, teams, and future modules—must belong to exactly one organisation. Users may belong to multiple organisations simultaneously and perform different roles within each organisation.

The architecture must satisfy the following objectives:

* complete tenant isolation;
* support multiple organisations per user;
* provide a simple and intuitive REST API;
* minimise the possibility of developer error leading to data leakage;
* simplify frontend implementation;
* avoid unnecessary authentication complexity;
* provide a scalable foundation for future modules.

Three approaches for resolving the active tenant were evaluated.

### Option 1 — URL Prefix

```text
/api/orgs/{org_slug}/tasks/
```

### Option 2 — Request Header

```text
X-Organisation-ID
```

### Option 3 — JWT Claim

```json
{
    "organisation": "..."
}
```

After evaluating maintainability, developer experience, debugging, security, and scalability, the URL-prefix approach was selected.

---

# Decision

The system will implement **shared-schema multi-tenancy**.

All organisations will share the same database while maintaining complete logical isolation through an `organisation` foreign key on every tenant-owned resource.

The active organisation will always be resolved from the request URL.

Example:

```text
GET /api/orgs/acme/tasks/
```

The organisation slug becomes the single source of truth for determining tenant context throughout the request lifecycle.

No organisation identifier will be stored inside authentication tokens.

No custom request headers will be required.

Changing organisations simply means navigating to another organisation URL.

# Multi-Tenancy Strategy

The application will use a **shared database** and **shared schema** architecture.

Every tenant-owned model will contain an `organisation` foreign key.

Example:

```text
Organisation
    │
    ├── Tasks
    ├── Projects
    ├── Files
    ├── Teams
    ├── Comments
    ├── Dashboards
    └── Future Modules
```

The only global model is:

```text
User
```

Users exist independently of organisations and become members through a Membership relationship.

---

# Domain Model

## Organisation

Represents an isolated tenant within the application.

Attributes:

* UUID primary key
* organisation name
* unique URL-safe slug
* owner
* created timestamp

Responsibilities:

* own all tenant resources;
* own memberships;
* own invitations;
* provide tenant isolation.

---

## Membership

Represents the relationship between a user and an organisation.

A user may belong to multiple organisations.

An organisation may contain many users.

Membership stores organisation-specific permissions.

Attributes:

* UUID
* organisation
* user
* role
* joined date

Supported roles:

```text
OWNER
ADMIN
MEMBER
```

---

## Invitation

Represents a pending invitation to join an organisation.

Invitations are independent of whether the invited email already has an account.

Attributes:

* UUID
* organisation
* invited email
* assigned role
* invitation token
* invited by
* accepted flag
* expiry timestamp (48 hours)
* creation timestamp

# API Structure

All tenant-aware resources will be nested beneath an organisation.

```text
POST   /api/orgs/
GET    /api/orgs/

GET    /api/orgs/{slug}/
PATCH  /api/orgs/{slug}/
DELETE /api/orgs/{slug}/

GET    /api/orgs/{slug}/members/
DELETE /api/orgs/{slug}/members/{user_id}/

POST   /api/orgs/{slug}/invitations/
GET    /api/orgs/{slug}/invitations/
DELETE /api/orgs/{slug}/invitations/{id}/

GET    /api/orgs/{slug}/tasks/
POST   /api/orgs/{slug}/tasks/
GET    /api/orgs/{slug}/tasks/{id}/
PATCH  /api/orgs/{slug}/tasks/{id}/
DELETE /api/orgs/{slug}/tasks/{id}/
```

Invitation endpoints remain public because invited users may not yet have an account.

```text
GET    /api/invitations/{token}/
POST   /api/invitations/{token}/accept/
POST   /api/invitations/{token}/register/
```

# Middleware

A dedicated organisation middleware will resolve the organisation from the URL.

Responsibilities:

* extract `{org_slug}`;
* retrieve the organisation;
* verify that it exists;
* attach it as `request.organisation`.

The middleware does **not** perform authorization or business logic.

Its only responsibility is resolving tenant context.

---

# TenantQuerysetMixin

All organisation-aware views will inherit from a shared base mixin.

Responsibilities:

* automatically filter every queryset by `request.organisation`;
* automatically assign the organisation when creating new records;
* eliminate duplicated filtering logic;
* prevent accidental cross-tenant queries.

Example behaviour:

```text
Task.objects.all()

↓

Task.objects.filter(
    organisation=request.organisation
)
```

Developers should never manually remember tenant filtering.

Tenant isolation becomes part of the framework rather than individual implementations.

---

# Authorization Model

Permissions are evaluated after authentication and organisation resolution.

Permission classes:

```text
IsOrgMember
IsOrgAdmin
IsOrgOwner
```

### OWNER

Can:

* delete organisation;
* transfer ownership;
* manage billing;
* manage administrators;
* perform all administrator actions.

### ADMIN

Can:

* invite members;
* remove members;
* manage organisation resources;
* create and edit tasks.

Cannot:

* delete organisation;
* transfer ownership.

### MEMBER

Can:

* access organisation resources;
* create tasks;
* update authorised resources.

Cannot:

* manage users;
* manage invitations;
* modify organisation settings.

# Invitation Architecture

The platform will support inviting users regardless of whether they already have an account.

Invitation lifecycle:

```text
Admin
   │
   ▼
Create Invitation
   │
   ▼
Generate Secure Token
   │
   ▼
Send Email
   │
   ▼
Frontend Invitation Page
   │
   ├──────────────┐
   ▼              ▼
Existing User   New User
   │              │
Login         Register
   │              │
   └──────┬───────┘
          ▼
Accept Invitation
          │
          ▼
Membership Created
          │
          ▼
Invitation Marked Accepted
```

Invitation tokens:

* expire after 48 hours;
* are single-use;
* become invalid immediately after acceptance.

Registration through an invitation creates the user account and organisation membership within a single database transaction to ensure consistency.

---

# Security Considerations

The architecture must guarantee tenant isolation.

Security measures include:

* organisation resolution through the URL;
* automatic queryset scoping;
* membership verification before resource access;
* role-based authorization;
* UUID identifiers;
* single-use invitation tokens;
* invitation expiry;
* transactional invitation acceptance;
* no organisation context stored inside JWTs.

---

# Consequences

## Positive

* Explicit tenant context in every request.
* Predictable REST API design.
* Easier debugging and logging.
* Simpler frontend implementation.
* No custom request headers.
* No JWT refresh when switching organisations.
* Automatic tenant isolation.
* Reduced likelihood of developer error.
* Easily extensible to future modules.
* Supports users belonging to multiple organisations.

## Negative

* Longer endpoint URLs.
* All tenant-aware routes require an organisation slug.
* Additional middleware execution for every organisation-scoped request.
* All future modules must include an organisation relationship where applicable.
