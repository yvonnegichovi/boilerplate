# ADR-0001: Git Branching Strategy

## Context

The project is a monorepo containing both the backend and frontend applications. Multiple contributors may work on different features simultaneously, requiring a branching strategy that supports parallel development while maintaining stable development and production environments.

The repository should provide:

- A stable production branch.
- A shared integration branch for ongoing development.
- Isolated feature development.
- A clear promotion path from development to production.
- Support for pull request reviews, automated testing, and CI/CD pipelines.

Without a defined branching strategy, contributors may introduce unstable code into production, create merge conflicts, or make collaboration difficult.

## Decision

The repository will adopt a simplified Git Flow branching strategy consisting of two long-lived branches and several short-lived branches.

### Long-lived branches

#### `main`

- Represents the production-ready codebase.
- Always contains stable, tested, and deployable code.
- Protected from direct commits.
- Changes are introduced only through Pull Requests from the `dev` branch.

#### `dev`

- Represents the primary integration branch.
- Serves as the staging branch.
- All completed features, bug fixes, and improvements are merged into this branch.
- Automatically deployed to the staging environment (when CI/CD is configured).

### Short-lived branches

All development branches are created from `dev` and merged back into `dev`.

Supported branch prefixes include:

| Prefix | Purpose |
|---------|---------|
| `feature/` | New functionality |
| `bugfix/` | Bug fixes discovered during development |
| `hotfix/` | Critical production fixes |
| `refactor/` | Code restructuring without changing functionality |
| `docs/` | Documentation updates |
| `chore/` | Tooling, dependencies, and maintenance |
| `test/` | Testing improvements |

Example:

```text
main
│
└── dev
    ├── feature/authentication
    ├── feature/dashboard
    ├── bugfix/login-error
    └── docs/update-readme
```

### Development workflow

1. Create a new branch from `dev`.
2. Implement the required changes.
3. Push the branch to the remote repository.
4. Open a Pull Request targeting `dev`.
5. Complete code review and automated checks.
6. Merge into `dev` after approval.
7. Perform integration and staging validation.
8. Merge `dev` into `main` when preparing a release.

### Hotfix workflow

Critical production fixes are branched from `main`.

After validation, the hotfix is merged into both:

- `main`
- `dev`

This ensures production fixes are preserved in future development.

### Branch protection

The following protections should be enabled for both `main` and `dev`:

- Require Pull Requests before merging.
- Require successful CI checks.
- Require at least one approving review.
- Prevent force pushes.
- Prevent branch deletion.

## Consequences

### Positive

- Provides a clear separation between development and production.
- Enables multiple developers to work independently.
- Keeps production stable by preventing direct commits.
- Simplifies code reviews through feature-specific Pull Requests.
- Supports continuous integration and future continuous deployment.
- Reduces merge conflicts by integrating work through a shared development branch.
- Makes releases predictable and easier to manage.

### Negative

- Requires contributors to follow the branching workflow consistently.
- Introduces an additional merge step when promoting releases from `dev` to `main`.
- Hotfixes require synchronization between both long-lived branches.
