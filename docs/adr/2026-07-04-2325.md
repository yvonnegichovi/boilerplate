# ADR-0002: Continuous Integration Workflow Strategy

## Context

The project is a full-stack monorepo containing a Django backend and a React (Vite) frontend. The repository follows a branching strategy where:

- `main` represents the production-ready codebase.
- `dev` serves as the integration and staging branch.
- Feature branches are created from `dev` and merged back into `dev`.
- Releases are promoted from `dev` to `main` through Pull Requests.

To support this workflow, an automated Continuous Integration (CI) pipeline is required to validate changes before they are merged. The pipeline should ensure that the backend and frontend can be built successfully while remaining flexible enough to accommodate future enhancements such as automated testing, linting, security scanning, and deployments.

Additionally, the CI configuration should evolve alongside the codebase and remain version-controlled within the repository.

## Decision

The project will use **GitHub Actions** as its Continuous Integration platform.

Two primary CI workflows will be maintained:

### Development CI (`ci-dev.yml`)

The Development CI workflow will run on Pull Requests targeting the `dev` branch.

Its responsibilities include:

- Installing backend dependencies.
- Running Django system checks.
- Installing frontend dependencies.
- Building the React application.

This workflow provides rapid feedback during feature integration while keeping execution time low.

### Production CI (`ci-main.yml`)

The Production CI workflow will run on:

- Pull Requests targeting `main`.
- Pushes to `main` after successful merges.

In addition to the validation performed by the Development CI workflow, it provides a dedicated production validation stage that can be extended in the future to include:

- Security scanning.
- Docker image builds.
- Deployment.
- Release creation.
- Notifications.

### Workflow Storage

Both workflow files will exist in the repository's `.github/workflows/` directory and will be present on both long-lived branches (`dev` and `main`).

Although each workflow exists in both branches, GitHub Actions determines when a workflow executes based on its configured event triggers (`on:`), not solely on the presence of the workflow file.

This ensures:

- CI configuration is version-controlled.
- Workflow changes undergo the same review process as application code.
- Development and production branches remain synchronized with respect to CI configuration.

## Consequences

### Positive

- Provides automated validation before code is merged.
- Ensures backend configuration and frontend builds remain functional.
- Supports the project's branching strategy by validating changes at both development and production stages.
- Keeps CI configuration under version control.
- Enables incremental enhancement of the CI pipeline as the project matures.
- Facilitates onboarding by providing a consistent and automated validation process.
- Reduces the likelihood of introducing build-breaking changes into shared branches.

### Negative

- CI workflows must be maintained alongside the application code.
- Separate workflows may introduce some duplication until reusable GitHub Actions or composite actions are adopted.
- Additional workflows may increase repository complexity as deployment and release automation are introduced.
